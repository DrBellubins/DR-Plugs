1. Top-Level Plugin Signal Flow

→

  • ReverbLeftRight->ProcessSample()

  • Uses GetDelayReverbGain(diffusionAmount)
  • At lower diffusion: delay dominates
  • At higher diffusion: reverb dominates

  • Processes blended wet signal
  • Intended to quantize pitch changes to echo boundaries
  • Design spec: never change pitch mid-echo
  • Affects both delay and reverb contributions after their blend

  • Receives dry + pitch-shifted wet
  • Returns distorted dry and distorted wet independently
  • Three serial distortion modules
  • Each module can target dry/wet/both depending on module DSP

  • Optional post HP/LP filter if filtersOrder == 2
  • Ducking uses dry input as detector
  • Ducking attenuates wet only
  • Distorted dry path is intentionally not ducked

  • distortionDry * dryVolume
  • duckedWet * wetVolume
  • Independent dry and wet level controls
  • No single crossfade mix knob

drySnapshot → pre-filter branch (optional, inside stage) → delay + reverb (parallel) → delay/reverb blend → pitch → distortion → optional post-filter → ducking → dry/wet gain sum → stereo

2. Chronoverb::ProcessBlock Detailed Execution Model

Per-block / per-sample model
Execution pattern

Per block

  • Dry input copied into drySnapshot
  • DelayLeftRight->ProcessBlock()
  • ReverbLeftRight->ProcessBlock()
  • PitchShifterLeftRight->ProcessBlock()
  • FilterLeftRight->ProcessBlock()
  • These allow deferred rebuilds / coefficient updates

Per sample

  • Read dry sample from snapshot
  • Optional pre-filter path for wet generators
  • Delay sample generation
  • Reverb sample generation
  • Blend, pitch, distort, filter, duck, sum, stereo
  • Write final L/R sample to output
Important architectural note: the current code computes the wet path from the preserved dry snapshot, not from already processed output. That is the correct structure for keeping stage interactions intentional.

Why this delay design matters for your stated goals:
  • Your design doc says the delay diffusion must create a swell into the nominal tap and smear both before and after it.
  • The current implementation achieves that with write diffusion plus early read diffusion.
  • The nominal tap and diffused early tap are blended, not abruptly swapped.
  • This is consistent with your “clean tap disappears by 0.5 diffusion amount” requirement.

Delay vs Reverb distinction in your implementation:
  • The delay stage preserves a nominal tap identity with diffusion around it.
  • The reverb stage is a more fully diffuse recirculating field.
  • Top-level mixing transitions between these behaviors using GetDelayReverbGain().
  • This matches your design requirement that the 0.5 → 1.0 region morphs from blurred echoes toward continuous reverb wash.

6. DiffusionChain Internal Model

Sample In → Stage 1 Allpass → Stage 2 Allpass → ... → Stage N Allpass → Sample Out
Allpass Network
Per-stage behavior
  • Each stage is a diffusion allpass with configurable delay and gain
  • Stage delays come from tuning tables interpolated by quality stage count
  • Diffusion size scales the tuning lengths
  • UpdateSize() slews the effective stage delays over time
  • This intentional slew causes the pitch-warp effect when size changes
Jitter
Micro-modulation
  • TPDF noise per stage
  • Low-pass filtered jitter state
  • Jitter depth in percent of base stage delay
  • Used to slightly animate the diffusion field
Design alignment
Why this is important relative to your design document
  • You explicitly require diffusion size smoothing that causes pitch warping; UpdateSize() is doing exactly that conceptually.
  • You explicitly require quality changes that do not shorten tails in a naive way; your quality-based interpolation of tunings supports that direction.
  • You explicitly require 8 carefully tuned allpass filters; current code supports up to 8 stages and uses tuning lists accordingly.

7. Pitch Shifter Placement and Conceptual Role

[Delay/Reverb Blended Wet] → Pitch Shifter → Distortion
Placement
Pitch operates after delay/reverb blend
  • Pitch does not process the untouched dry input directly in Chronoverb
  • It processes the already-generated wet field
  • This means both delay-like and reverb-like content are pitch-processed together
Design intent
Echo-boundary pitch changes
  • Your instructions require pitch to change only at echo boundaries
  • Attached pitch code includes boundary-aware interfaces: OnEchoBoundary(), SetInitialRatio()
  • This is aligned with your requirement not to retune mid-echo
Sequence modes
Implemented sequence building blocks
  • Progressive up/down style logic
  • Ping-pong octave sequence
  • Random octave sequence
  • Granular backend prepared for quantized boundary changes

9. Filter Stage Placement

10. Ducking Stage

Dry Input Detector → Envelope Follower → Duck Control = Envelope × Duck Amount → Wet Gain Reduction
Detector
Dry-Signal Peak Detection
  • Uses max(abs(L), abs(R))
  • Boosted internally for sensitivity
→
Envelope
Attack / Release Smoothing
  • Fast-ish rise, slower fall based on parameters
  • Realtime envelope follower
→
Gain Control
Wet-Only Attenuation
  • wetGain = 1 - (detectorEnvelope * duckAmount)
  • Dry signal remains unaffected
  • Distorted dry path remains intact

11. Stereo Stage

If spread < 0: narrow toward mono | If spread > 0: Haas-like widen by delaying derived mid into right side
Negative spread
Stereo Narrowing
  • Crossfades both channels toward mono sum
  • Useful for collapsing image
Positive spread
Haas-style Widening
  • Builds mid = 0.5 * (L + R)
  • Reads delayed mid from delay line
  • Blends delayed mid into right channel
  • Intended as widening stage, though code comment notes it needs improvement

13. Design-Constrained Interpretation

Important
How the current implementation aligns with your ChronoverbInstructions
  • Delay diffusion swell into the tap: supported by early-read diffusion plus blended nominal tap.
